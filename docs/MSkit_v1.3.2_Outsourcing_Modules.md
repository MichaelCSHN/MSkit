# MSkit v1.3.2 外协分包方案（三方叙事 + 一级包 + OP 工作包映射）

> **基线**：`MSkit_v1.3.2_Spec.md` + `MSkit_v1.3.2_Positioning_and_Scenarios.md` + v1.3 明细文档  
> **状态**：内部评审稿 / 外协规划建议  
> **目的**：在组织 / 搜索 / 防护三方智能规控平台定位下，将 OP-01–OP-12 工作分解合并为更适合发包管理的“6 个一级外包包 + 1 个内部核心包”，并明确 MVP 路演阶段、正式工程阶段与场景脚本的外协边界。

---

## 1. 总体结论

MSkit 可以并行外协开发，但不应把 OP-01–OP-12 直接拆成 12 个独立合同。OP 层级适合作为内部 WBS 与验收颗粒度；对外发包应按能力和耦合关系合并为 6 个一级包。

三方叙事下，外协的目标不是分别外包“组织方、搜索方、防护方”，而是外包支撑三方平台的能力模块：地图/定位态势/报告、目标检测/变化检测、覆盖规划、设备平台、硬件机电、RF/认证和制造测试。组织/搜索/防护角色边界、演示脚本、合规措辞和最终产品责任必须由内部团队保留。

推荐执行结构：

- **内部核心包（不外包）**：产品定位、三方角色定义、场景脚本、系统架构、接口冻结、合规边界、安全架构、数据模型、供应商协调、最终集成与验收。
- **A. Rugged Hardware & Mechatronics**：结构、电源、电池、热设计、环境预验证。
- **B. RF / Wireless / Certification**：地区化无线 SKU、天线、射频预扫描、认证路径。
- **C. Edge Platform & Device Firmware**：Jetson BSP、系统镜像、OTA、LoRa 低速遥测、RTK/GNSS/IMU 融合。
- **D. Application Software**：组织方 定位态势/规划、地图、任务、报告、Dashboard、防护方 覆盖规划 UI、三方角色视图。
- **E. AI Data & Model Optimization**：搜索方 目标检测/变化检测、数据集、标注、训练、量化、TensorRT/DeepStream 模型层优化。
- **F. Manufacturing & Test Fixtures**：试产、工装、出厂测试、良率、FRU/售后流程。

> **发包规则**：对外 SOW / RFP 使用一级包 A–F；OP-01–OP-12 只作为二级工作包、验收项和内部跟踪项。未经 CCB 批准，不单独把 OP 拆成独立合同。

## 2. MVP 路演阶段与外协关系

MVP 路演演示是 M0 预研/融资阶段，不作为正式外协一级包发包。建议由内部 Product / Full-stack / AI 小组主导，必要时只采购小范围短期支持。

| MVP 工作 | 场景 / 三方归属 | 建议归属 | 是否正式外协包 |
|----------|----------------|----------|----------------|
| Hide and Seek 主脚本 | 组织/搜索/防护完整闭环 | 内部 Product / Demo Owner | 否 |
| 野外搜救备选脚本 | 组织/搜索，可选防护方安全区 | 内部 Product / Demo Owner | 否 |
| 路演叙事、演示脚本、合规边界 | 组织方主视角 + 全局叙事 | 内部核心包 | 否 |
| 笔记本 Web App、地图、报告、定位态势、路径规划 | 组织方能力 + 通用能力 | D 包能力的小范围原型支持 | 不建议签长期 D 包 SOW，可用短期 Sprint |
| 防护方 覆盖规划、保护区、禁入区、覆盖缺口、巡查路径 | 防护方能力 | D 包能力的小范围原型支持 | 不建议签长期 D 包 SOW，可用短期 Sprint |
| 搜索方 目标检测/变化检测 演示模型 | 搜索方能力 | E 包能力的小范围原型支持 | 不建议签完整 E 包 SOW，可用短期模型/标注任务 |
| 无人机视频/图片/日志导入 | 通用数据接入 | 内部 + C/D 能力支持 | 否，MVP 不做飞控 |
| 演示素材采集 | Hide and Seek / 野外搜救素材 | 内部或本地飞手/素材服务 | 否 |

MVP 阶段不得让外协方实现或展示无人机飞控控制、自动航线下发、军事化目标分类、武器识别、打击排序或作战控制能力。正式 A–F 外协包应在 M1 规格冻结后按 SOW 启动。

## 3. 内部核心包（不外包）

| 内部保留事项 | 原因 | 内部 Owner |
|--------------|------|------------|
| 产品定位与目标场景 | Hide and Seek、Milsim、安保训练、野外搜救等场景涉及对外叙事和合规风险 | PM / LEG |
| 组织/搜索/防护角色定义 | 角色边界错误会导致产品混乱和合规误解 | PM / Product Architect |
| 路演脚本与 Demo 叙事 | MVP 是融资/立项材料，不能由外协随意发挥 | PM / Demo Owner |
| 系统架构与接口冻结 | 所有外包包依赖统一接口；接口失控会导致集成失败 | System Architect |
| 安全架构 | LUKS、TPM、Secure Boot、RBAC、审计、加密导出涉及核心信任边界 | EMB Lead / FS Lead |
| 数据模型与 API 契约 | 前端、AI、地图、报告、Mesh、定位融合都依赖统一 schema | FS Lead |
| 端到端集成 Owner | 供应商只能证明模块达标，整机指标必须内部签署 | QA / Integration |
| 供应商选择与合规判断 | 热成像、无线、电池、开源许可证、隐私均涉及公司责任 | SCM / LEG |
| 控制类无人机集成 | v1.3.2 不纳入 MVP；如未来开展，需单独安全与合规评审 | PM / LEG / EMB |

## 10. 接口治理规则

1. 所有一级包必须基于冻结的 API schema、数据模型、硬件接口图、无线 SKU 矩阵、三方角色权限和测试标准开发。
2. 供应商不得单方面变更组织/搜索/防护角色边界、接口、无线频段、电池方案、数据格式、坐标系、时间同步或安全策略。
3. 接口变更必须提交 CCB，说明影响范围、兼容性、测试计划和回滚策略。
4. 所有交付物必须入库：源代码、构建脚本、原理图、CAD、BOM、测试日志、报告、许可证清单。
5. 外协交付必须可复现：内部团队应能在本地重新构建、重新刷写、重新测试。
6. SOW / RFP 必须同时引用 v1.3.2 覆盖层和对应 v1.3 明细文档，并列明适用章节；不得只引用单一文件。
7. MVP 阶段的短期外协不得引入飞控控制、自动航线下发、军事化目标分类、武器识别、打击排序或作战控制功能。
