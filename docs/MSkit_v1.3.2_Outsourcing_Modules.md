# MSkit v1.3.2 外协分包并行开发方案

> **基线**：`MSkit_v1.3.2_Spec.md`  
> **状态**：内部评审稿 / 外协规划建议  
> **目的**：将 MSkit 拆分为可并行外协的工作包，同时明确内部保留事项、接口边界、验收条件和依赖关系。

---

## 1. 总体结论

MSkit 可以拆分为多个外协分包并行开发，但不建议整体外包。项目复杂度来自硬件、嵌入式、AI、通信、结构、电源、合规、制造和现场测试的交叉耦合。正确拆分方式是：

- **内部保留系统架构、接口冻结、合规边界、核心安全、集成验收和最终产品责任**；
- **外协承担边界清晰、交付物可验收、可替换的模块**；
- **每个外协包必须有接口文档、样机/代码/报告交付物、测试方法和 Go/No-Go 条件**；
- **外协模块之间不得直接私下改接口，所有接口变更必须走 CCB**。

---

## 2. 不建议外包的内部核心事项

| 内部保留事项 | 原因 | 内部 Owner |
|--------------|------|------------|
| 产品需求与合规边界 | 涉及民用定位、禁用功能、宣传口径、法律责任 | PM / LEG |
| 系统架构与接口冻结 | 所有分包依赖统一接口；接口失控会导致集成失败 | CTO / System Architect |
| 安全架构 | LUKS、TPM、Secure Boot、RBAC、审计、加密导出涉及核心信任边界 | EMB / FS |
| 数据模型与 API 契约 | 前端、AI、地图、报告、Mesh 都依赖统一 schema | FS |
| 最终集成与验收签署 | 外协只能证明模块达标，不能替代整机责任 | PM / QA |
| 供应商选择与出口/认证判断 | 热成像、无线、电池均涉及地区合规与责任 | SCM / LEG |
| 无人机控制类集成 | v1.3.2 不纳入 MVP；如未来做，需单独安全与合规评审 | PM / LEG / EMB |

## 3. 推荐外协分包结构

### OP-01 结构与工业设计分包

| 项目 | 内容 |
|------|------|
| 范围 | Pelican 1450/1550 内部布局、支架、泡沫、面板、散热风道、开孔、防水穿舱件、肩带/推车附件 |

| 分包类型 | 合同方式 | 付款建议 |
|----------|----------|----------|
| 结构/热/电源/RF | 固定范围 SOW + 里程碑验收 | 30% 启动 / 40% 样件或报告 / 30% 验收 |

| 前端/地图/报告 | Sprint + API 合同测试 | 每 2–3 周验收一次可运行版本 |
| 认证/法务/SBOM | 专项咨询 + 报告验收 | 报告交付后付款，证书另计 |
| 试产/工装 | NRE + 单台加工/组装费 | NRE 按治具/SOP 验收，单台按出货结算 |

## 6. 接口治理规则

1. 所有外协包必须基于冻结的 API schema、数据模型、硬件接口图和测试标准开发。
2. 外协方不得单方面变更接口、无线频段、电池方案、数据格式或安全策略。
3. 接口变更必须提交 CCB，说明影响范围、兼容性、测试计划和回滚策略。
4. 所有交付物必须入库：源代码、构建脚本、原理图、CAD、BOM、测试日志、报告、许可证清单。
5. 外协交付必须可复现：内部团队应能在本地重新构建、重新刷写、重新测试。

## 7. 最小内部团队配置建议

即使采用外协，内部至少需要保留：

| 角色 | 人数 | 责任 |
|------|------|------|
| PM / Product Owner | 1 | 需求、分包管理、预算、里程碑、客户试点 |
| System Architect / Tech Lead | 1 | 架构、接口、技术取舍、集成问题裁决 |
| Embedded Lead | 1 | Jetson、BSP、驱动、系统服务、硬件联调 |
| Full-stack Lead | 1 | API、数据模型、Dashboard 集成、报告系统 |
| AI Lead | 1 | 数据、模型、评估、推理优化验收 |
| QA / Integration | 1（可兼职） | 验收脚本、测试计划、缺陷追踪、出厂测试 |
| Legal/Compliance | 外部顾问 + 内部 PM 对接 | 合规、许可证、用户协议、认证资料 |

## 8. 外协风险与控制措施

| 风险 | 影响 | 控制措施 |
|------|------|----------|
| 接口失控 | 模块无法集成 | API/硬件接口冻结，变更走 CCB |
| 外协只交 Demo 不交可维护资产 | 后续无法量产 | 合同写明源码、脚本、CAD、BOM、测试日志、文档必须交付 |
| AI 指标不可复现 | 验收争议 | 冻结测试集、固定 benchmark 条件、内部复测 |
| RF/电池地区合规遗漏 | 认证失败或无法运输 | OP-04/OP-11 前置，地区化 SKU 明确 |
| 结构、电源、热设计互相冲突 | 返工 | M2 前完成机械/电源/热联合评审 |
| 供应商锁定 | 成本失控 | 每个关键器件至少 1 个替代方案 |
| 安全实现外包后不可审计 | 数据与密钥风险 | 安全架构内部设计，外协实现必须代码审查和渗透测试 |

## 9. 推荐第一批外协优先级

优先启动：

1. OP-01 结构与工业设计。
2. OP-02 电源、电池与热插拔。
3. OP-04 RF、地区化 SKU 与认证预审。
4. OP-06 AI 数据集、训练与 TensorRT 优化。
5. OP-11 合规、隐私与开源许可证审计。

暂缓外包或仅做小范围 PoC：

1. 无人机控制类集成。
2. 核心安全架构。
3. 系统 API 与数据模型设计。
4. 整机最终集成与验收。

---

## 10. 一句话执行建议

MSkit 应采用“内部小核心团队 + 多外协并行模块”的模式推进。最适合外协的是结构、电源、热设计、RF/认证、AI 数据、前端、地图报告、LoRa 低速遥测、试产工装和合规审计；最不适合外协的是系统架构、接口治理、安全边界、合规判断和最终验收。
